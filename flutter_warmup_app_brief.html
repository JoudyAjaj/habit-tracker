Warm-up App Brief — Flutter Refresh

Pre-project preparation · 2 weeks · Read this before writing a single line of code

The app: Personal Habit Tracker

A simple mobile app to create daily habits, mark them as done each day, and see a streak count. No backend, no login — all data stored locally using shared_preferences. Small enough to finish in 2 weeks. Complex enough to touch the widgets, state, and navigation you'll need on the real project.

Why this app

It's deliberately limited. The goal is to rebuild your Flutter muscle memory, not to ship something impressive. Every screen in it maps directly to patterns you'll use on the freelance project:

  • A list of items → the real project has listing screens
  • Adding/editing a form → the real project has create/edit flows
  • Simple local state management → the real project uses Provider or Riverpod
  • Navigation between screens → the real project uses GoRouter

Screens to build

Screen 1Habit List
Shows all habits. Each item displays the habit name and today's checkbox. Tapping the checkbox marks it done for today and updates the streak count. A floating action button opens Screen 2.
Screen 2Add / Edit Habit
A form with a text field for the habit name and a save button. Validates that the name is not empty and not a duplicate. Used for both creating and editing — the same screen, pre-filled when editing.
Screen 3Habit Detail
Shows one habit's full history: current streak, longest streak, and a simple 7-day grid showing which days this week were completed. Has a delete button with a confirmation dialog.

Features list

  • Add a new habit with a name
  • Mark a habit as done for today (once per day only)
  • Streak counter — consecutive days completed
  • Edit a habit name
  • Delete a habit with a confirmation dialog
  • Persist all data locally with shared_preferences
  • Empty state on the list screen when no habits exist yet

Build phases

1
Days 1–3 · Static UI only
Build all three screens as layout-only widgets. No state, no navigation, hardcoded fake data. Focus on widgets: Column, Row, ListView, Card, TextField, FloatingActionButton, AlertDialog.
2
Days 4–6 · Navigation
Connect the screens with Navigator 2.0 or GoRouter. Pass a habit object from the list to the detail screen. Make the FAB navigate to Add screen. Make edit work by passing existing data into the same form screen.
3
Days 7–9 · State management
Introduce a HabitProvider using Provider (or Riverpod if you prefer). Move all habit logic out of widgets into the provider. Marking done, streak calculation, add/edit/delete — all in the provider, widgets just read and display.
4
Days 10–12 · Local persistence
Add shared_preferences. Serialize your habit list to JSON and write it on every change. Read it on app startup. App should survive being killed and reopened with data intact.
5
Days 13–14 · Polish and self-review
Handle edge cases: empty state, duplicate name validation, what happens if you mark a habit done twice. Read your own code out loud. Can you explain every function? Fix anything you can't explain.

AI usage rules — read carefully

These rules apply for the entire 2 weeks. They are not suggestions.

Allowed

Ask AI to explain a concept you don't understand — "explain how Provider works", "what is null safety in Dart", "explain what setState does"

Allowed

Ask AI to explain code you already wrote — paste your code and ask "what does this do, and is there a problem with it?"

Allowed

Ask AI to review your logic — "here is my streak calculation, is this correct?"

Allowed

Ask AI about Flutter widget syntax when you know what you want to build but forgot the exact API — "how do I make a ListTile's trailing show a Checkbox?"

Not allowed

Asking AI to generate a whole screen or feature — "build me the Add Habit screen". If you use it, you skip the learning.

Not allowed

Copy-pasting AI code without reading it. If you copy something, you must be able to delete it and rewrite it yourself before moving to the next phase.

Not allowed

Using AI to fix bugs for you — "my code crashes, here it is, fix it". Instead: "my code crashes, here is the error, what should I look at?"

Not allowed

Generating the data model or provider class using AI. The HabitModel and HabitProvider are the core of this app — write them yourself, even if it takes hours.

The one rule that overrides everything

At the end of every day, pick any function you wrote that day and explain out loud what it does, line by line — as if you're explaining it to someone who has never seen code. If you can't do that, you don't move to the next phase until you can.

What the mentor will check

  • Can you explain why you put the state in the provider instead of the widget?
  • What happens to the streak if the user skips a day?
  • Why did you choose this widget over another option?
  • Walk me through what happens when the app opens — from main() to the first screen rendered.

These are the same kinds of questions you'll get in a real code review on the freelance project. This app is the practice round.
